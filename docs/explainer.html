🔐 auth-security-poc

A visual walk-through of the two authentication patterns in this repo — JWT refresh-token rotation with reuse detection and the OAuth 2.1 Authorization Code flow with PKCE — plus the scaling story when you run more than one replica.

One app, several hats

A single Spring Boot 3 / Java 21 process acts as authorization server, resource server, and a demo public client — so you can see every side of the handshake in one place. Everything is in-memory, so it boots clean every run.

POC 1

JWT refresh rotation

Short-lived stateless access tokens (5 min) + long-lived opaque refresh tokens (7 days) that rotate on every use. Replaying an old refresh token trips reuse detection and revokes the whole token family.

POC 2

OAuth 2.1 + PKCE

Authorization Code flow for a public client, where a PKCE S256 challenge binds the party that starts the flow to the party that redeems the code — no client secret needed.

POC 1 — JWT access tokens + refresh-token rotation

The hard problem: a JWT is stateless, so you can verify it anywhere — but you can't take it back. Keep access tokens short; make the long-lived refresh token stateful, single-use, and self-policing.

Two tiers, opposite trade-offs

Play the lifecycle

Pick a scenario, then step through it.

The tripwire, in code

public RefreshToken rotate(String presented) {
    RefreshToken existing = tokensById.get(presented);
    if (existing == null)    throw new RefreshTokenException("Unknown refresh token");
    if (existing.revoked())  throw new RefreshTokenException("Refresh token revoked");
    if (existing.isExpired())throw new RefreshTokenException("Refresh token expired");

    if (existing.used()) {                       // ← reuse = proof of theft
        revokeFamily(existing.familyId());       //   burn the whole family
        throw new RefreshTokenException("Refresh token reuse detected; family revoked");
    }
    existing.markUsed();                          // single-use
    return mint(existing.userId(), existing.familyId());   // successor, same family
}

Legit clients always present the newest token. A thief who copied an earlier one eventually presents an already-used token → the tripwire fires and ejects everyone in the family, forcing a clean re-login.

POC 2 — OAuth 2.1 Authorization Code + PKCE

The hard problem: a public client (SPA / mobile) can't keep a secret, and the authorization code travels through the interceptable browser. PKCE proves the redeemer is the initiator — with no client secret.

The handshake

Only the hash (challenge) crosses the interceptable front-channel. The secret (verifier) is revealed only in the back-channel token POST — and only after the code is spent. An intercepted code is useless without it.

Constant-time compare (the silent one)

public static boolean verify(String verifier,
        String challenge, String method) {
  if (!"S256".equals(method)) return false;   // no 'plain'
  String computed = s256Challenge(verifier);
  return MessageDigest.isEqual(               // ← constant time
      computed.getBytes(US_ASCII),
      challenge.getBytes(US_ASCII));
}

String.equals short-circuits on the first mismatched byte — a timing side-channel that lets an attacker brute-force the challenge one byte at a time. MessageDigest.isEqual takes the same time regardless.

Tech debt — honest about the shortcuts

  • Tokens/codes are in-memory, raw — production stores a hash in Redis/Postgres.
  • Signing key sits in application.yml — inject via env / KMS / Vault.
  • HS256 single shared secret — prefer RS256/EdDSA + JWKS when verification is split across services.
  • rotate() is check-then-act, not atomic — needs a DB transaction or Lua script under concurrency.
  • No consent UI, no OIDC id_token, no MFA — use Keycloak / Ory for a real IdP.

Scaling — what breaks when you add a second replica

The access token scales for free (any replica verifies a JWT from the key). The stateful parts — refresh rotation, reuse detection, single-use codes — live in one JVM's heap and break the moment traffic spreads across pods / VMs.

The dangerous, silent break

The fix: stateless pods + shared store + atomic mutations

Refresh tokens (hashed)Postgres / RedisUPDATE … WHERE used=false RETURNING or Lua

The service APIs (issueNewFamily, rotate, revokeFamily, consume) stay identical — only the backing store changes. Pass the load test at replicas ≥ 3 and you're done: the guarantee must hold regardless of which replica serves each request. Full detail in CONSISTENCY.md.
